# ADR-0005 — Origins and TLS: named Cloudflare Tunnel is the production origin; quick tunnels support passkeys

**Status:** Accepted (2026-07-17) — F-06 ruling · **Revised 2026-07-22** — F-06a (quick-tunnel passkeys)
**Finding trail:** F-06, F-06a
**Requirements:** `REQUIREMENTS.md` §2 (tunneling rows), §5.1

## Context

WebAuthn passkeys, the Screen Wake Lock API (kitchen display and table displays), and dependable audio autoplay all require a secure context (HTTPS). Passkeys additionally bind to a **relying-party identifier (RP ID) derived from the origin's host**, so the origin must be stable **for the lifetime of a given passkey** — i.e. between the run on which it is registered and the runs on which it is used. The original requirements were silent on certificates and leaned on Cloudflare quick tunnels for demos.

Two facts shape the design:

1. **`trycloudflare.com` is on the Public Suffix List.** Each quick tunnel gets a random subdomain, and because the parent is a public suffix, that random subdomain **is** the registrable domain. A passkey registered through a quick tunnel binds to that one-run hostname; it works perfectly **for the life of that tunnel** and stops working when a *new* run hands out a *different* random subdomain. There is no way to make a quick-tunnel passkey survive a URL change — but that is a *re-registration on the next demo*, not a reason passkeys cannot be exercised on a quick tunnel at all.
2. Installing a private certificate authority root on **guests' phones** is not a real option, so any "internal CA on the LAN" origin can only ever serve staff-owned devices, never the guest ordering flow.

The owner's GoTunnels project served as the working reference for tunnel lifecycle, TLS posture, observability wiring, **and — decisively for F-06a — for running a full passkey (indeed passkey-only) flow over a `try.cloudflare.com` quick tunnel.** GoTunnels does this by deriving the WebAuthn relying party **per request** from the browser's origin when it matches a trusted `https://*.trycloudflare.com` pattern, rather than pinning it at boot. Its session model — a never-expiring bearer token in `localStorage` — was examined and **explicitly rejected**: it does not fit Blazor Server's cookie-plus-circuit authentication model and would undermine the security-stamp revalidation this system relies on.

## Decision

1. **Production origin = a persistent, named Cloudflare Tunnel on the operator's stable domain.** This is a **persistence and operability** choice — a stable public URL for QR join codes, bookmarks, and passkeys that must outlive individual runs — **not** a precondition for passkeys to function. `RESTAURANT_PUBLIC_ORIGIN` is that `https://` origin; it drives QR join URLs (ADR-0009) and is the canonical/primary trusted WebAuthn origin. Guests, staff devices, and table displays all use this origin — including devices physically inside the restaurant.
2. **The WebAuthn RP ID is derived per request, not pinned at boot (F-06a).** `IdentityPasskeyOptions.ServerDomain` is left **null**, so the .NET 10 passkey handler uses `Request.Host.Host` for the RP ID on each ceremony. `PublicOriginMiddleware` normalizes `Request.Host` to the browser's real origin host (from the unforgeable `Origin` header when trusted, else the configured public origin), and `IdentityPasskeyOptions.ValidateOrigin` gates the browser's cryptographically-signed origin against a trust set: the configured `RESTAURANT_PUBLIC_ORIGIN`, any `RESTAURANT_TRUSTED_ORIGIN_PATTERNS` entry (default `https://*.trycloudflare.com`), and loopback in development. This makes passkeys **self-healing** across tunnel URL changes and — the point of F-06a — makes them work on a quick tunnel without the random URL being known at startup. It is safe because credentials are RP-ID-scoped by the authenticator itself (a page on `attacker.trycloudflare.com` can only exercise credentials created for `attacker.trycloudflare.com`); the origin trust set is defence in depth. The patterns gate **only** WebAuthn relying-party derivation — the app is single-origin and adds no cross-origin CORS allowance, so the CORS-with-credentials hazard that would make this unsafe does not arise.
3. **Topology:** `cloudflared` (compose `production` profile) connects outbound to Cloudflare and forwards to `web` over the private compose network. TLS terminates at the Cloudflare edge; the tunnel leg is encrypted by `cloudflared`. Caddy is not in the production request path.
4. **Development origin:** Caddy terminates TLS for `https://localhost:8443` with its internal CA (compose default profile), giving developers a secure context with zero ceremony.
5. **Optional staff LAN fallback (documented, off by default):** Caddy may additionally serve a LAN hostname with its internal CA, roots installed on **staff-owned** devices only. During a WAN outage this keeps kitchen/counter/administration reachable — by password + TOTP, or by a passkey previously registered on that same LAN origin (a passkey registered on the public origin is bound to that host and will not assert on the LAN origin). Guest ordering and table displays are down during a WAN outage regardless.
6. **Quick tunnels (`try.cloudflare.com`) are for demos, and passkeys work on them (revised).** `scripts/quick_tunnel.sh` brings the stack up, opens the tunnel, discovers the `*.trycloudflare.com` URL, sets `RESTAURANT_PUBLIC_ORIGIN` to it so QR join URLs and the form-post fallback are correct, and holds the tunnel in the foreground. Within a run you can register and sign in with a passkey — including a passkey-only account (ADR-0010 already permits one with no password). The script prints, prominently, the one caveat that remains: a **new run gets a new random URL**, so passkeys (and any bookmarks) from a previous run will not carry over and must be re-registered. Password + TOTP remains the durable baseline for accounts that must survive URL changes. **A quick tunnel must still never carry the *bootstrap* of a real, long-lived instance** — bootstrap on the stable named-tunnel domain so the first administrator's credentials persist.

## Consequences

- **Accepted risk (owner-acknowledged): in-house ordering hairpins through Cloudflare.** A guest sitting in the restaurant reaches the server in the same building via the Cloudflare edge; LAN ordering therefore depends on WAN health. This is the price of one stable public origin without touching guest devices.
- **Passkeys are no longer coupled to a named tunnel.** The old ruling ("passkeys require the stable named-tunnel domain; quick tunnels are password+TOTP-only") is **superseded** by decision 2. Demos can showcase the passwordless flow; the only quick-tunnel limitation is non-persistence of the URL across runs.
- Rootless Podman cannot bind ports 80/443 by default; the stack publishes 8080/8443, and the operations guide offers the `net.ipv4.ip_unprivileged_port_start=80` sysctl for installations that want standard ports for the LAN fallback. The tunnel path does not need inbound ports at all.
- Rotating join tokens (ADR-0009) embed `RESTAURANT_PUBLIC_ORIGIN`; changing the domain invalidates any passkeys registered on the old host (WebAuthn, not a bug) and re-points join URLs. Domain changes are a documented, disruptive operation for a persistent instance (operations guide §9); on a quick tunnel it simply happens every run, which is why quick tunnels are for demos.

## History

- 2026-07-16 — drafted as "Caddy internal CA on LAN + named tunnel for remote".
- 2026-07-17 — reworked to tunnel-as-primary-origin after the F-06 ruling and Public Suffix List verification; GoTunnels adopted as reference, its bearer-token session model rejected.
- 2026-07-22 — **F-06a revision.** Corrected the claim that passkeys require the named-tunnel domain. Per-request RP derivation (`ServerDomain=null`) + `PublicOriginMiddleware` + `ValidateOrigin` against `RESTAURANT_TRUSTED_ORIGIN_PATTERNS` (default `https://*.trycloudflare.com`) make passkeys work on quick tunnels, matching the GoTunnels reference. The named tunnel remains the production origin for persistence, not as a passkey prerequisite.
